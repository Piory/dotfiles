# Platform UI/UX Design

## 適用トリガー
次の依頼が来たらこのスキルを適用する。
- 「このURLを参考にUI/UXを作って」
- 「このページを参考にデザインシステムを作って」
- 「既存デザインを改修して」
- 「プラットフォーム別に最適化したUI/UX案を出して」

## 目的
- 参照URLのデザインをそのまま複製せず、設計原則に抽象化して再利用する。
- 対象プラットフォームに合わせたUI/UXへ落とし込む。
- `デザインシステム作成` と `既存デザイン改修` の両方に対応する。

## 事前確認（不足があれば確認する）
1. 参照URL（1件以上）
2. 対象プラットフォーム（`web` / `ios` / `ipados` / `macos` / `android` / `cross-platform`）
3. タスク種別（`design-system` または `redesign`）
4. 制約（ブランドカラー、実装技術、納期、アクセシビリティ要件）

## プラットフォーム別の参照スキル
- `web`: `web-design-guidelines`（必須） + `frontend-design`（実装や表現設計が必要な場合）
- `ios`: `ios-design-guidelines`
- `ipados`: `ipados-design-guidelines`
- `macos`: `macos-design-guidelines`
- `android`: `android-design-guidelines`
- `cross-platform`: 上記を組み合わせ、共通基盤は `ui-ux-pro-max` の原則も参照する

補足:
- この `platform-ui-ux-design` スキル経由では、`ui-ux-pro-max` の Python スクリプト実行を許可する。
- `ui-ux-pro-max` の Python スクリプトを実行しない場合は、`SKILL.md` の設計原則のみ参照する。

## 実行フロー
1. 参照URLを分析する
   - 情報設計、画面構成、余白、タイポグラフィ、カラー、コンポーネント、状態遷移、モーションを整理する。
   - 事実（観測できる内容）と推測（意図・背景）を分ける。
2. プラットフォーム要件を重ねる
   - 参照スキルを使って、各OS/プラットフォームの慣習・制約・アクセシビリティ要件を適用する。
3. 成果物を作る
   - `design-system` の場合: デザイントークン、コンポーネント仕様、状態定義、レイアウト規則、アクセシビリティ基準を定義する。
   - `redesign` の場合: 現状課題、優先度（P0/P1/P2）、改修方針、変更後仕様、影響範囲を定義する。
4. 妥当性チェック
   - 一貫性、可読性、操作性、アクセシビリティ、実装可能性を確認する。

## 重要な制約
- 参照URLのデザインを直接コピーしない（構造・原則に抽象化する）。
- 画像・ロゴ・文言などの著作物を無断で流用しない。
- 不明点は断定せず、必要に応じて「わかりません」または「（推測です）」で明示する。
- URLへアクセスできない場合は、スクリーンショットや追加情報を依頼する。

## 完了条件
- タスク種別に応じた成果物（`design-system` か `redesign`）がある。
- プラットフォーム別の適用方針が明示されている。
- 参照分析と最終提案の対応関係が説明できる。
