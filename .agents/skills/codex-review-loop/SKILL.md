# Codex Review Loop

## 目的

`codex review` の結果を鵜呑みにせず、レビュー実行エージェントが各指摘を調査して対応要否を判断する。対応が必要な指摘だけを Herdr の専用ペインにいる実装担当エージェントへ直接依頼し、完了後に差分と検証結果を確認して再レビューする。

このスキルは Claude Code と Codex CLI のどちらから呼び出されても同じ手順で動作する。実行環境固有のツール名や、自分を Claude または Codex と決めつける表現は使用しない。

終了条件はレビュー出力を機械的にゼロにすることではなく、**事実確認できた対応必須・対応推奨の指摘が残っていないこと**とする。

## 役割分担

| 役割 | 責務 |
|---|---|
| レビュー実行エージェント | 対象決定、`codex review` 実行、指摘の事実確認、ユーザー確認、差分と検証結果の確認、最終報告 |
| レビュー用 Codex | 読み取り専用サンドボックスでコードレビューを行い、指摘を返す |
| 実装担当エージェント | Herdr の専用ペインで確認済みの指摘だけを修正し、実装の最後に `simplify` スキルを使用してから検証結果を報告する |

レビュー実行エージェントは修正を直接実装しない。実装担当へ委譲し、設計判断と検証を分離する。

## 必須原則

- すべてのレビューを `codex --sandbox read-only review` で実行し、レビュー用 Codex にファイルを書き込ませない。
- レビュー出力は調査の起点であり、そのまま修正指示として扱わない。
- 指摘ごとに対象コード、呼び出し元、関連テスト、仕様、現在の差分との因果関係を確認する。
- 誤検知、防御済み、主観的提案、現在の差分と無関係な問題は実装担当へ依頼しない。
- 判断材料が不足する場合は推測で進めず、追加調査後も不明ならユーザーへ質問する。
- 質問には、実行環境およびプロジェクト指示で指定された専用の質問機能を使用する。特定製品の質問ツール名を前提にしない。
- 実装依頼は構造化したプロンプトとして直接渡し、依頼用の一時ファイルや仕様ファイルを作成しない。
- 1 回の実装依頼は 1 目的に限定し、無関係な指摘をまとめない。
- 実装担当は修正実装の最後に `simplify` スキルを使用し、今回変更したコードだけを挙動を変えずに整理する。最終検証は `simplify` 適用後に実行する。
- ユーザーが明示しない限り、コミット、push、PR/MR 作成を行わない。
- 既存変更や並行して発生した変更を勝手に戻さない。認知していない変更を検出した場合は停止してユーザーへ確認する。
- リポジトリ内で適用される `AGENTS.md`、`CLAUDE.md`、関連スキルを確認し、より具体的な指示を優先する。

## 0. 前提確認

最初に利用可能な `herdr` スキルを読み、現在の CLI に合った操作方法を確認する。その後、次を実行する。

```bash
test "${HERDR_ENV:-}" = "1"
test -n "${HERDR_PANE_ID:-}"
test -n "${HERDR_TAB_ID:-}"
command -v codex
command -v herdr
command -v jq
git rev-parse --show-toplevel
```

Herdr 管理下でない場合や必須コマンドがない場合は、別ペインへの安全な委譲ができないため停止し、不足している前提を報告する。ペイン ID の解析には `jq` を使用し、Python は使用しない。

開始時の状態を記録する。

```bash
git status -sb
git status --porcelain=v1
git diff --stat
git diff --cached --stat
```

この記録を、実装担当が対象外の変更を加えていないか確認する基準にする。

## 1. レビュー対象を決定する

### 未コミット差分がある場合

staged、unstaged、untracked のいずれかがあれば、未コミット差分を対象にする。

```bash
git status --porcelain=v1
codex --sandbox read-only review --uncommitted
```

### 未コミット差分がない場合

現在のブランチに対応する PR/MR の base branch、またはユーザーが明示した base branch を使用する。base branch はリポジトリに適した連携機能や CLI で事実確認し、`main`、`master`、`develop` などを推測しない。

```bash
BASE_BRANCH=<事実確認した base branch>
git rev-parse --verify "${BASE_BRANCH}^{commit}"
codex --sandbox read-only review --base "$BASE_BRANCH"
```

PR/MR が存在せず base branch を確認できない場合は、専用の質問機能でユーザーへ確認する。

### Codex CLI の制約

`--uncommitted`、`--base`、`--commit`、独自 `PROMPT` は互いに競合するため、レビュー対象フラグと独自プロンプトを併用しない。バグ、回帰、セキュリティ、データ不整合、仕様逸脱などに限定する品質基準は、後続の指摘検証工程で適用する。

レビューコマンドが失敗した場合は原因を確認して一度だけ再試行する。再度失敗したらループを停止し、実行コマンド、確認できた原因、未確認事項を報告する。

## 2. 指摘を調査して対応要否を判定する

各指摘について、次の順番で確認する。

1. 指摘された行と周辺の実装を読む。
2. 入力元、呼び出し元、戻り値の利用先まで追跡する。
3. 関連する型、DB 制約、認可、例外処理、既存テストを確認する。
4. 仕様書、Issue、ADR、コメントなどに期待動作があれば照合する。
5. 問題が成立する具体的な入力、状態、並行実行条件を説明できるか確認する。
6. 現在の差分が問題を新規に発生させたか、既存問題を悪化させたか確認する。
7. 必要に応じて対象テストや最小の再現手順を実行する。

重大度はレビュー出力に記載された値を保持し、記載がなければ推測して補わない。

| 判定 | 基準 | 次の操作 |
|---|---|---|
| 対応必須 | バグ、回帰、セキュリティ、データ破損、仕様逸脱を事実確認できた | 実装担当へ依頼 |
| 対応推奨 | 明確な保守性・性能リスクがあり、今回の差分内で安全に修正できる | 実装担当へ依頼 |
| 対応不要 | 誤検知、防御済み、主観的提案、対象外、または修正リスクが上回る | 理由を記録し依頼しない |
| 判断不能 | 追加調査後も仕様または成立条件を確定できない | ユーザーへ質問 |

最低限、指摘ごとに次を記録する。

```text
指摘:
確認した事実:
成立条件と影響:
判定: 対応必須 / 対応推奨 / 対応不要 / 判断不能
判定理由:
修正対象:
触らない範囲:
```

循環する指摘では、文言ではなく「対象ファイル、原因、成立条件、実害」を過去の指摘と比較する。ユーザーが既に受容したトレードオフを、理由なく再実装して往復させない。

## 3. 各レビュー結果を表示・保持する

レビューを実行するたびに、指摘の調査と判定が完了した直後、その回の結果をユーザーへ表示する。実装依頼や次のレビューまで表示を遅らせない。指摘なしの回やレビュー実行失敗時も省略しない。

```md
## Codex Review #<回数> サマリー

- レビュー対象: `<--uncommitted / --base branch / --commit SHA>`
- Review 結果: pass / 指摘あり / 実行失敗
- 指摘数: <件数。重大度別件数は出力で確認できる場合のみ記載>

| # | 重大度 | 指摘概要 | 確認した事実 | 判定 | 次の操作 |
|---|---|---|---|---|---|
| 1 | P1 | ... | 成立条件と影響 | 対応必須 | 実装担当へ依頼 |

集計: 対応必須 <件数> / 対応推奨 <件数> / 対応不要 <件数> / 判断不能 <件数>
次の操作: <実装依頼 / ユーザー確認 / 再レビュー / 終了>
```

各回のサマリーはレビュー回数と対応付けて保持し、最終報告で同じ順序と判定内容のまま再掲する。後続調査で判定が変わった場合は、元の判定を書き換えず変更理由を追記する。

対応必須・対応推奨が 0 件なら実装担当ペインを作成せず、サマリー表示後に残存リスクを報告して終了する。

## 4. 必要な場合だけユーザーへ質問する

次の場合は、実装依頼の前にユーザーへ確認する。

- 追加スコープや別機能が必要になる。
- 過去の判断を覆す、または実装を往復させる。
- 仕様、データモデル、互換性、性能、コストに実質的なトレードオフがある。
- 対応範囲を事実確認だけでは決定できない。
- 認知していない変更が見つかった。

質問時は推奨案を先頭にし、各選択肢の影響を簡潔に示す。明確なバグ修正で範囲と期待動作を確定できる場合は、不要な確認を挟まない。

## 5. 実装担当ペインを準備する

対応必須・対応推奨の指摘がある場合だけ、現在のペインを右に分割して専用の Codex を起動する。同じループ中は作成したペインを再利用する。ユーザーまたは依頼元が明示的に指定していない既存ペインを勝手に使用しない。

```bash
REPO_ROOT=$(git rev-parse --show-toplevel)
WORKER_PANE=$(herdr pane split "$HERDR_PANE_ID" --direction right --no-focus | jq -r '.result.pane.pane_id')
test -n "$WORKER_PANE"
test "$WORKER_PANE" != "null"
herdr pane run "$WORKER_PANE" "codex -C \"$REPO_ROOT\" --no-alt-screen"
herdr wait agent-status "$WORKER_PANE" --status idle --timeout 30000
```

分割や起動が失敗した場合はペイン ID を推測せず、`herdr pane list` と `herdr pane read` で状態を確認する。同じペインを無反応のまま繰り返し作成しない。

受信側が別ペインからの依頼であることと、レビューの実施主体を判別できるよう、次の識別文を先頭にそのまま含める。

## 7. 質問と完了を監視する

```bash
herdr wait output "$WORKER_PANE" \
  --match 'REVIEW_LOOP_(QUESTION|DONE):' \
  --regex \
  --source recent \
  --timeout 1800000
herdr pane read "$WORKER_PANE" --source recent-unwrapped --lines 160
```

`wait output` のタイムアウトや agent status の変化だけで完了と判断しない。必ずペイン出力を読み、質問、権限承認、完了、失敗のどれかを判別する。

実装担当から質問を受けた場合の扱いは次のとおりとする。

| 質問の種類 | 対応 |
|---|---|
| コード、仕様、ドキュメントから事実確認できる | レビュー実行エージェントが確認して同じペインへ回答する |
| 製品判断、仕様変更、複数の妥当な選択肢がある | 専用の質問機能でユーザーへ確認してから回答する |
| 権限昇格、破壊的操作、外部への書き込み | リスクと必要性を確認し、自動承認しない |
| 認知していない変更 | 勝手に戻さずユーザーへ確認する |

回答は `herdr pane run` で直接送る。TUI の選択操作が必要な場合だけ `herdr pane send-text` と `herdr pane send-keys` を使用する。画面状態を確認せずに `Enter` や `Ctrl-C` を送らない。

## 8. 変更と検証結果を確認する

実装担当の完了報告だけで成功扱いにしない。

1. `git status -sb` と `git diff` で開始時からの変更範囲を確認する。
2. 対象外または認知していない変更があれば、勝手に戻さずユーザーへ確認する。
3. 指摘された成立条件が解消されたことをコード上で確認する。
4. 実装担当が `simplify` スキルを使用し、今回変更した範囲だけを挙動を変えずに整理したことを確認する。
5. 追加・変更されたテストが問題の再現条件を検証しているか確認する。
6. リポジトリの指示に従い、対象テスト、型検査、lint、必要なら build を実行する。
7. 検証失敗時は原因を調査し、必要な修正だけを同じペインへ直接再依頼する。

## 9. 再レビューして収束させる

最初に決定した対象方式で `codex --sandbox read-only review` を再実行し、新しい指摘にも調査と対応要否判定を最初から適用する。再レビュー前に、対象となる `git diff` に実装担当の変更が含まれることを確認する。

レビュー対象から変更が漏れる場合は、自動でコミットして対象を変えない。レビュー可能な状態にする方法をユーザーへ確認する。

次のいずれかに達するまで反復する。

- 事実確認できた対応必須・対応推奨の指摘が 0 件になった。
- 同じ実質的な指摘が 2 回連続し、追加修正では前進しない。
- 5 回のレビュー・修正サイクルに達した。
- 想定外の変更、仕様判断、外部要因により安全に継続できない。

残りがユーザー受容済みのトレードオフや、理由を確認した対応不要の指摘だけなら完了扱いとする。際限のない微最適化を目的にループを続けない。

## 完了報告

ループ終了時は、最新回だけでなく、各回で表示したレビューサマリーを時系列で再掲する。最終報告は次の順番とする。

1. **最終結果**: 完了、停止、またはユーザー判断待ちと、その理由。
2. **全体集計**: レビュー回数、検出指摘数、対応必須・推奨の修正数、対応不要数、未解決数。
3. **レビュー履歴**: `Codex Review #1` から最終回までのサマリー。
4. **実装・検証結果**: 実装担当へ依頼した内容、確認した差分、実行したテスト・型検査・lint。
5. **残存事項**: 対応不要の理由、見送り合意済みの指摘、未解決事項、残存リスク。
6. **Git 操作**: コミット、push、PR/MR 作成の実施有無。明示指示がなければ未実施と記載する。

報告を簡潔にする目的で、各回のサマリーや対応不要とした理由を省略してはならない。

## 互換性要件

- レビュー実行には Codex CLI の `codex review` を使用し、特定ホストのプラグイン、ラッパースクリプト、バージョン付き絶対パスへ依存しない。
- シェル実行、バックグラウンド実行、質問は、呼び出し元が提供する機能を使用し、特定環境だけの API 名や表記を手順に含めない。
- 環境固有のリンク記法や、特定プロジェクト専用の検証コマンドを含めない。
- Herdr と Codex CLI のコマンド構文は、実行前に各 CLI の `--help` と利用可能なスキルで確認する。
