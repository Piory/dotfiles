# Implementation Workflow

## 目的

ユーザーの実装要求を、仕様が曖昧なまま実装へ渡さない。親エージェントが次の工程を統括し、専門の名前付き subagent に仕様策定、実装、レビュー修正を委譲する。

1. `implementation-planner` が既存コードとドメイン文書に基づいて仕様を固める。
2. 親エージェントが仕様を精査して承認する。
3. `implementation-agent` が承認済み仕様だけを実装する。
4. 親エージェントが差分、検証結果、仕様適合を確認する。
5. 親エージェントが `codex review` を実行して各指摘を事実確認する。
6. `review-fixer` が確認済みの対応対象だけを修正する。
7. 対応すべき指摘がなくなるまで、確認、修正、再レビューを繰り返す。

このスキルは Codex の名前付き subagent 機能を前提とする。Herdr や別ペインは使用しない。

## 役割

| 役割 | 責務 | 禁止事項 |
|---|---|---|
| 親エージェント | ユーザーとの対話、subagent の起動と継続、仕様承認、差分検証、レビュー指摘の事実確認、最終報告 | 実装とレビュー修正を直接行わない |
| `implementation-planner` | `grill-with-docs` を使った調査、質問の作成、実装仕様と受け入れ条件の確定 | ファイル変更、実装、ユーザーへの直接質問 |
| `implementation-agent` | 承認済み仕様の実装、テスト、`simplify`、検証 | 仕様変更、`codex review`、別 subagent への委譲 |
| レビュー用 Codex | 読み取り専用で現在の実装差分をレビュー | ファイル変更 |
| `review-fixer` | 親が確認した指摘の最小修正、テスト、`simplify`、検証 | 未確認指摘の修正、仕様変更、`codex review` |

親エージェントは調査や検証を行ってよいが、コード変更を伴う実装とレビュー修正を自分で行ってはならない。subagent が失敗した場合も、親が代わりに実装せず、再依頼または停止理由の報告を行う。

## 必須原則

- subagent は `agent_type` に `implementation-planner`、`implementation-agent`、`review-fixer` のいずれかを明示して起動する。
- 各工程では同じ目的に対して生成済みの subagent スレッドを継続利用し、質問への回答だけで新しい agent を増やさない。
- 書き込み可能な subagent は同時に複数動かさず、必ず直列に実行する。
- subagent への依頼は、必要な文脈を含む構造化プロンプトとして直接渡す。一時的な仕様ファイルを作ってパスだけを渡さない。
- すべての subagent の質問は親エージェントが受け取り、コードや文書から確認できる事実は親が調査して回答する。
- 製品判断や複数の妥当な選択肢が必要な場合だけ、親が実行環境の専用質問機能でユーザーへ一度に1問ずつ確認する。
- subagent の完了報告を鵜呑みにせず、親が実際の差分と検証結果を確認する。
- `codex review` の出力は調査の起点とし、親が成立条件と影響を確認する前に修正へ渡さない。
- ユーザーが明示しない限り、コミット、push、PR/MR作成を行わない。
- 既存変更や並行変更を勝手に戻さない。認知していない変更を検出した場合は停止し、ユーザーへ確認する。
- リポジトリ内で適用される `AGENTS.md`、関連スキル、ユーザー指示を確認し、より具体的な指示を優先する。

## 0. 前提確認

最初に次を確認する。

```bash
command -v codex
git rev-parse --show-toplevel
git status -sb
git status --porcelain=v1
git diff --stat
git diff --cached --stat
```

Codex の subagent 一覧で次の名前付き agent が利用可能であることを確認する。

- `implementation-planner`
- `implementation-agent`
- `review-fixer`

いずれかを利用できない場合は、親が代替実装せず、不足している agent と確認方法を報告して停止する。

開始時の状態を記録し、今回の変更と既存の未コミット変更を区別する基準にする。認知していない変更がある場合は、その時点でユーザーへ確認する。

## 1. 実装要求を整理する

親エージェントはユーザーの要求を改変せず、planner へ渡すために次を整理する。

```text
実装要求:
期待する成果:
既知の制約:
対象として明示された範囲:
対象外として明示された範囲:
関連するユーザー判断:
開始時の Git 状態:
```

要求が短い、または曖昧でも親の推測だけで仕様を補完しない。コードや文書を調査して解決できる事項を含め、まず planner に渡す。

## 2. `implementation-planner` に仕様策定を依頼する

名前付き subagent を `agent_type = "implementation-planner"` で起動する。依頼にはユーザー要求、制約、既知の判断、開始時の Git 状態を含める。

```text
## 目的
ユーザー要求を、実装担当へ渡せる検証可能な仕様にしてください。

## 必須手順
- `grill-with-docs` スキルを使用する。
- 既存コード、CONTEXT.md、CONTEXT-MAP.md、ADR、関連ドキュメントを調査する。
- コードや文書から確認できる事項は自分で確認する。
- ユーザー判断が必要な質問は親エージェントへ一度に1問だけ返す。
- ファイルを変更しない。

## 入力
<手順1で整理した内容>

## 完了条件
未解決事項がなく、対象範囲、対象外、実装方針、境界条件、テスト方法、受け入れ条件が具体化されていること。
```

planner の応答を次のマーカーで判定する。

| マーカー | 親の操作 |
|---|---|
| `IMPLEMENTATION_PLAN_QUESTION:` | 質問の必要性と推奨回答を精査し、事実調査またはユーザー確認後、同じ planner へ回答する |
| `IMPLEMENTATION_PLAN_DONE:` | 手順3で仕様を精査する |
| マーカーなし | 必要な形式と未完了事項を示し、同じ planner へ再報告を依頼する |

planner から複数の質問が同時に返った場合は、依存順に1問へ絞るよう同じ planner へ依頼する。

## 3. 親エージェントが仕様を精査する

親エージェントは、次の条件をすべて満たすまで仕様を承認しない。

| 観点 | 確認内容 |
|---|---|
| 要求整合 | ユーザーの目的と期待する成果を変えていない |
| 根拠 | 現在の挙動と変更理由がコードまたは文書の事実に基づく |
| 境界 | 対象範囲、対象外、変更しない既存挙動が明確 |
| 完全性 | 正常系、異常系、境界条件、互換性が必要な範囲で定義済み |
| 実装可能性 | 変更候補と依存関係が具体的で、実装担当が推測を必要としない |
| 検証可能性 | テスト方法と受け入れ条件が観測可能 |
| 安全性 | 既存変更、データ、権限、外部副作用へのリスクが整理済み |
| スコープ | 無関係な機能追加やリファクタリングが含まれていない |

コードや文書から解消できる不足は親が調査し、その事実を同じ planner に返して仕様を更新させる。ユーザー判断が必要なら専用質問機能で確認し、回答を同じ planner に返す。

承認した仕様は親のコンテキストで保持し、実装中に黙って変更しない。仕様変更が必要になった場合は planner の工程へ戻る。

## 4. `implementation-agent` に実装を依頼する

仕様承認後、`agent_type = "implementation-agent"` で名前付き subagent を起動する。

```text
この依頼は implementation-workflow の承認済み実装依頼です。親エージェントが同スキルと codex review を実施するため、この subagent では実施しないでください。

## 実装目的
<承認済みの目的>

## 承認済み仕様
<手順3で承認した仕様全文>

## 変更条件
- 対象外を変更しない。
- 既存の未コミット変更を戻さない。
- 認知していない変更や仕様矛盾を検出したら停止して質問する。
- コミット、push、PR/MR作成を行わない。

## 検証
<承認済みのテスト・検証方法と受け入れ条件>
```

応答の扱いは次のとおりとする。

| マーカー | 親の操作 |
|---|---|
| `IMPLEMENTATION_QUESTION:` | 事実調査またはユーザー確認を行い、同じ implementation-agent へ回答する |
| `IMPLEMENTATION_DONE:` | 手順5で実装を精査する |
| マーカーなし | 完了条件と報告形式を示し、同じ implementation-agent へ再報告を依頼する |

質問への回答が承認済み仕様を変える場合は、実装を継続させず planner の工程へ戻る。

## 5. 親エージェントが実装を精査する

実装担当の報告だけで完了扱いにしない。親エージェントが次を確認する。

1. `git status -sb` と `git diff` で開始時からの変更範囲を確認する。
2. 対象外または認知していない変更がないことを確認する。
3. 変更された実行経路を読み、承認済み仕様と受け入れ条件に対応していることを確認する。
4. エラー処理、境界条件、互換性、外部副作用に認識齟齬がないことを確認する。
5. 追加・変更されたテストが仕様の再現条件と期待結果を検証していることを確認する。
6. `simplify` が今回の変更だけに適用され、挙動を変えていないことを確認する。
7. 必要なテスト、型検査、lint、buildを親側でも実行または結果確認する。

仕様逸脱や実装漏れがある場合は、根拠と期待動作を同じ `implementation-agent` に返して修正を依頼する。親が直接修正してはならない。

仕様そのものに問題が見つかった場合は planner の工程へ戻る。認知していない変更が見つかった場合は勝手に戻さず、ユーザーへ確認する。

## 6. `codex review` を実行する

手順5を通過し、今回の実装差分が未コミット状態に含まれていることを確認してから、読み取り専用でレビューする。

```bash
codex --sandbox read-only review --uncommitted
```

`--uncommitted` と独自プロンプトを併用しない。開始時から存在する未コミット変更もレビュー対象に含まれるため、親は各指摘が今回の実装に起因するかを必ず確認する。

レビューコマンドが失敗した場合は原因を確認して一度だけ再試行する。再度失敗したらループを停止し、実行コマンド、確認した原因、未確認事項を報告する。

各レビュー回について、対象、結果、指摘数、各指摘の判定、次の操作を保持する。

## 7. レビュー指摘を事実確認する

各指摘について、対象コード、呼び出し元、関連テスト、承認済み仕様、現在の差分を調査する。問題が成立する具体的な入力、状態、実行経路、影響を説明できない指摘を修正へ渡さない。

| 判定 | 基準 | 操作 |
|---|---|---|
| 対応必須 | バグ、回帰、セキュリティ、データ破損、仕様逸脱を事実確認できた | `review-fixer` へ依頼 |
| 対応推奨 | 明確な保守性・性能リスクがあり、今回の範囲で安全に解消できる | `review-fixer` へ依頼 |
| 対応不要 | 誤検知、防御済み、主観的提案、既存問題、今回の対象外 | 理由を記録して修正しない |
| 判断不能 | 調査後も仕様または成立条件を確定できない | ユーザーへ質問 |

最低限、指摘ごとに次を記録する。

```text
指摘:
確認した事実:
成立条件と影響:
今回の実装との因果関係:
判定:
判定理由:
修正対象:
変更しない範囲:
検証方法:
```

重大度はレビュー出力に記載された値を保持し、記載がなければ推測して補わない。

## 8. `review-fixer` に確認済み指摘を依頼する

対応必須または対応推奨がある場合だけ、`agent_type = "review-fixer"` で名前付き subagent を起動する。同じレビュー修正ループでは同じ fixer を継続利用し、書き込み可能な別 agent と同時実行しない。

1回の依頼は1目的に限定する。独立した指摘は同じ fixer へ順番に渡す。

```text
この依頼は implementation-workflow のレビュー修正依頼です。親エージェントが codex review と指摘の事実確認を実施済みです。この subagent では再レビューせず、確認済み指摘だけを修正してください。

## 承認済み仕様
<承認済み仕様の関連部分>

## 確認済み指摘
<指摘、重大度、成立条件、影響、今回の実装との因果関係>

## 修正条件
<期待動作、修正対象、変更しない範囲>

## 検証
<再現手順、対象テスト、型検査、lint、build>
```

| マーカー | 親の操作 |
|---|---|
| `REVIEW_FIX_QUESTION:` | 反証や質問を調査し、必要ならユーザー確認後、同じ fixer へ回答する |
| `REVIEW_FIX_DONE:` | 差分と検証結果を親が確認する |
| マーカーなし | 完了条件と報告形式を示し、同じ fixer へ再報告を依頼する |

fixer が指摘は成立しないと報告した場合は、親が根拠を再確認する。成立しなければ判定を対応不要へ変更し、変更理由をレビュー履歴へ追記する。

## 9. 修正確認と再レビューを繰り返す

fixer の完了後、親が次を確認する。

1. 指摘の成立条件が解消されている。
2. 承認済み仕様と既存挙動を壊していない。
3. 変更範囲が依頼した指摘に限定されている。
4. テストが再現条件と修正結果を検証している。
5. `simplify` 後の検証が成功している。

問題があれば同じ fixer に根拠付きで再依頼する。問題がなければ手順6へ戻り、修正を含む未コミット差分を `codex review` で再レビューする。

次のいずれかに達するまで手順6から9を繰り返す。

- 事実確認できた対応必須・対応推奨の指摘が0件になった。
- 同じ実質的な指摘が2回連続し、追加修正で前進しない。
- 5回のレビュー・修正サイクルに達した。
- 認知していない変更、仕様判断、外部要因により安全に継続できない。

上限や停止条件に達した場合は完了扱いにせず、未解決事項と必要な判断を報告する。対応不要と事実確認した指摘だけが残る場合は、理由を明記して完了できる。

## 10. 最終報告

最終報告は次の順とする。

1. **最終結果**: 完了、停止、またはユーザー判断待ちと理由。
2. **承認仕様の要約**: 目的、対象範囲、対象外、受け入れ条件。
3. **実装内容**: 変更ファイルと主要な挙動。
4. **検証結果**: テスト、型検査、lint、build、`simplify` の結果。
5. **レビュー全体集計**: レビュー回数、総指摘数、修正数、対応不要数、未解決数。
6. **レビュー履歴**: 各回の指摘、確認した事実、判定、修正内容、再レビュー結果。
7. **残存事項**: 対応不要の理由、未解決事項、残存リスク。
8. **Git操作**: コミット、push、PR/MR作成の実施有無。

ユーザーが確認できるよう、実行していない検証を成功と記載せず、事実と推測を区別する。
